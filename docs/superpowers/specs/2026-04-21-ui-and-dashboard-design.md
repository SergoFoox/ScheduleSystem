# Специфікація дизайну: Інтерфейс та Дашборд (ASMS V3)

**Дата:** 2026-04-21
**Статус:** На розгляді
**Технологічний стек:** Vaadin 25.1.2 (Hilla/React), Spring Boot 4.0.5, Timefold 1.33.0, Signals (@preact/signals-react).

## 1. Огляд
Цей документ описує дизайн та архітектуру користувацького інтерфейсу для Системи автоматизованого управління розкладом (ASMS V3). Основна увага приділяється створенню динамічного, реактивного дашборда для диспетчера з можливостями швидкого коригування розкладу.

## 2. Архітектура інтерфейсу (Hilla + React)

### 2.1. Структура сторінок
*   **Маршрутизація**: Використовується файлова маршрутизація Hilla (`src/main/frontend/views/`).
*   **Головний макет (`@layout.tsx`)**: 
    *   Бічне меню (Side Navigation) з використанням Aura Theme.
    *   Верхня панель (Header) з перемикачем статусів «Чернетка/Опубліковано».
*   **Головна сторінка (`@index.tsx`)**: Дашборд з сіткою розкладу.

### 2.2. Комунікація з бекендом
*   Використання `@BrowserCallable` ендпоінтів у Java для передачі DTO.
*   Автоматична генерація TypeScript клієнтів для безпечної типізації.

## 3. Модуль: Дашборд (Schedule Grid)

### 3.1. Функціональність сітки
*   **Гнучкі осі**: Перемикання між режимами перегляду:
    *   *За групами*: Стовпці — Групи, Рядки — Таймслоти.
    *   *За викладачами*: Стовпці — Викладачі, Рядки — Таймслоти.
    *   *За аудиторіями*: Стовпці — Аудиторії, Рядки — Таймслоти.
*   **Drag-and-Drop**: Можливість перетягування занять між комірками. При кожній дії надсилається запит на сервер для валідації жорстких обмежень (Hard Constraints).
*   **Візуалізація конфліктів**: Комірки з порушеннями підсвічуються червоним (`aura-error-color`).

### 3.2. Реактивність (Signals)
*   Використання `Signal` для зберігання поточного стану обраного об'єкта (групи чи викладача).
*   При кліку на заняття у сітці, сигнал оновлює вміст бічної панелі аналітики без перезавантаження сторінки.

## 4. Модуль: Бічна панель аналітики (Analytics Sidebar)

### 4.1. Вміст панелі
*   **Віджет навантаження**: Візуалізація виконання навчального плану для поточної групи/викладача.
*   **Пошук вікон**: Список вільних таймслотів («вікон») для оптимізації розкладу.
*   **Попередження**: Список м'яких обмежень (Soft Constraints), які наразі порушені (наприклад, забагато занять на день).

## 5. Модуль: «Швидкі заміни» (Smart Replacements)

### 5.1. Алгоритм підбору
При активації режиму заміни для конкретного заняття:
1.  **Фільтр за компетенцією**: Система відбирає викладачів, які мають право викладати дану дисципліну (`TeacherCompetenceMatrix`).
2.  **Фільтр за часом**: Перевірка, чи має викладач «вікно» у цей час.
3.  **Пріоритезація**: Виведення списку кандидатів з позначенням їхньої завантаженості.

### 5.2. Користувацький інтерфейс
*   Діалогове вікно (`Vaadin Dialog`) зі списком вільних викладачів.
*   Кнопка «Призначити заміну», яка миттєво змінює викладача у `Lesson` та перераховує розклад.

## 6. Безпека та Керування Версіями

### 6.1. Статуси розкладу
*   **Чернетка (Draft)**: Доступна лише диспетчеру. Дозволяє вільно експериментувати з Timefold Solver.
*   **Опубліковано (Published)**: Версія, доступна для перегляду студентами та викладачами. Редагування заборонено до створення нової чернетки.

### 6.2. Ролі (Spring Security)
*   `ROLE_DISPATCHER`: Повний доступ до всіх модулів та функцій редагування.
*   `ROLE_USER`: Тільки перегляд опублікованого розкладу.
