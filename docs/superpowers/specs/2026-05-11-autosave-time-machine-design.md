# Дизайн-специфікація: Машина часу (Автозбереження системи)

**Дата**: 2026-05-11
**Статус**: Очікує на розгляд
**Мета**: Впровадження механізму автоматичного фонового збереження всього стану системи (викладачі, групи, плани, уроки) для запобігання втраті даних та можливості швидкого відкату змін.

## 1. Огляд архітектури

Функція складається з трьох основних частин:
1.  **Сховище знімків**: Спеціальна таблиця в БД для зберігання JSON-серіалізованого стану системи.
2.  **Фоновий агент (Scheduler)**: Компонент бекенда, що виконує знімки за розкладом.
3.  **Інтерфейс "Машина часу"**: UI-компонент для перегляду та відновлення даних.

## 2. Технічні деталі

### 2.1. Модель даних (Backend)
Створення нової сутності `AutosaveSnapshot`:
- `Long id` (PK)
- `LocalDateTime timestamp` (Час створення)
- `String snapshotData` (Повний JSON знімок)
- `Integer entityCount` (Для статистики в UI)

### 2.2. Алгоритм знімка
Кожні 2 хвилини бекенд виконує наступне:
1.  Збирає всі записи з таблиць: `Teacher`, `StudentGroup`, `Room`, `Subject`, `CoursePlan`, `Lesson`.
2.  Використовує `ObjectMapper` для створення цілісного JSON-об'єкта.
3.  Зберігає новий запис у `AutosaveSnapshot`.
4.  **Ротація**: Видаляє найстаріші записи, залишаючи лише останні **5 штук**.

### 2.3. Логіка відновлення
Ендпоінт `restoreSnapshot(Long id, boolean asNewTemplate)`:
- Якщо `asNewTemplate == true`: Створює новий запис у `SavedSchedule` на основі даних знімка. Поточні дані в робочих таблицях не змінюються.
- Якщо `asNewTemplate == false`: Виконує повне очищення робочих таблиць та імпорт даних зі знімка (повний відкат).

## 3. Дизайн інтерфейсу (Frontend)

### 3.1. Кнопка доступу
- Розміщується на панелі інструментів `DashboardView`.
- Стиль: `theme="tertiary"`, іконка `vaadin:clock`. Текст: "Історія".
- Повинна гармонійно вписуватися в існуючу синю палітру Aura Theme.

### 3.2. Діалог історії
Використовується стандартний `vaadin-dialog`:
- Список карток із мітками часу (наприклад, "Сьогодні о 14:02").
- При виборі знімка відкривається підтвердження.

### 3.3. Вікно підтвердження
- Текст: "Відновити стан системи від [Час]?"
- Компонент `Checkbox`: "Створити новий розклад як копію" (ввімкнено за замовчуванням).
- Кнопка "Відновити" (синя, `theme="primary"`).

## 4. План тестування
- Перевірка спрацювання таймера (Scheduled task).
- Перевірка ліміту в 5 записів у БД.
- Тестування відновлення:
    - Як нового шаблону (чи з'явився у списку).
    - Як повного відкату (чи змінилися дані в сітці розкладу).

## 5. Ризики
- Об'єм даних у JSON може бути значним при дуже великій кількості груп (100+). Для поточної версії це допустимо, але може знадобитися стиснення в майбутньому.
