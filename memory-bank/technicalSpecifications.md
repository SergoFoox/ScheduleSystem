# Технічне завдання: Система автоматизованого управління розкладом (V3)

## 1. Загальний огляд
Даний документ описує вимоги до розробки вебзастосунку для автоматизації створення та управління навчальним розкладом. Система спрямована на оптимізацію роботи диспетчера, забезпечення актуальною інформацією студентів та викладачів, а також інтелектуальну підтримку прийняття рішень при заміні занять.

---

## 2. Основні сутності та мапінг (Data Model)
Для функціонування алгоритмів автоматизації необхідно впровадити наступну структуру даних:

### 2.1. Таблиця «Викладач — Предмет» (Матриця компетенцій)
| Поле | Опис |
| :--- | :--- |
| **ПІБ викладача** | Ідентифікатор викладача та приналежність до кафедри. |
| **Перелік дисциплін** | Дисципліни, які викладач має право та компетенцію читати. |
| **Пріоритетність** | Позначка «основний лектор» або «тільки практика». |
| **Тип заняття** | Лекція, Практика, Лабораторна робота. |

### 2.2. Таблиця «Навчальний план курсу»
| Поле | Опис |
| :--- | :--- |
| **Курс/Група** | Назва навчальної групи або потоку. |
| **Дисципліна** | Назва та загальний обсяг годин на семестр. |
| **Розподіл годин** | Кількість лекцій та практик на тиждень. |
| **Вимоги до приміщення** | Специфікація: лабораторія, комп'ютерний клас, лекційна аудиторія. |

## 3. Функціональні можливості

### 3.1. Підтримка циклічності (Чисельник/Знаменник)
* **Параметр періодичності:** Додавання атрибута для кожного заняття (щотижня, по парних, по непарних тижнях).
* **Логіка відображення:** Алгоритм має коректно розташовувати пари «раз на два тижні» в одній часовій комірці без конфліктів.

### 3.2. Контроль версій (Режим «Чернетка»)
* **Статус «Чернетка»:** Розклад доступний лише диспетчеру для редагування.
* **Статус «Опубліковано»:** Фінальна версія стає видимою для всіх користувачів.

### 3.3. Облік навантаження (Workload)
* **Тижневий ліміт:** Контроль максимальної кількості годин викладача на тиждень (N годин).
* **Семестровий баланс:** Відстеження відсотка виконання навчального плану.
* **Типи ставок:** Обмеження днів перебування в університеті для сумісників.

---

## 4. Модуль «Автоматичний розклад» (The Engine)

### 4.1. Жорсткі обмеження (Hard Constraints)
* Відсутність часових накладок для викладача, групи або аудиторії.
* Кількість студентів у групі ≤ Місткість аудиторії.

### 4.2. М'які обмеження (Soft Constraints)
* **Мінімізація «вікон»:** Компактне розташування пар.
* **Побажання по часу:** Врахування зручності викладачів.
* **Рівномірність:** Збалансоване навантаження за днями тижня.

---

## 5. Додаткові модулі

### 5.1. Управління аудиторним фондом
* Облік місткості та обладнання (проектор, ПК, спеціальні верстати).
* Врахування географічного розташування (корпуси) та часу на переміщення між ними.

### 5.2. Інтелектуальний модуль «Швидкі заміни»
* Автоматичний підбір вільних колег з відповідними компетенціями у разі відсутності основного викладача.
* Оновлення розкладу та балансу годин в «один клік».

### 5.3. Аналітика, звіти та експорт
* **Генерація звітності:** Автоматичне створення звітів за формою 2-П та статистикою навантаження.
* **Експорт розкладу:** Можливість завантаження персонального розкладу (для групи або викладача) у форматах **HTML** та **PDF**.
* **Аналіз фонду:** Статистика використання аудиторій.

## 6. Технічні вимоги

### 6.1. Стек технологій
* **Backend:** Java.
* **Frontend:** Vaadin (забезпечує багатий користувацький інтерфейс на Java).
* **Optimization Engine:** OptaPlanner / Timefold (для вирішення завдань планування та оптимізації розкладу).

### 6.2. Інтерфейс та Дашборди
* **Платформа:** Вебзастосунок (Responsive Web Design).
* **Дашборд:** Візуалізація невичитаних годин по кожному предмету/викладачу.
